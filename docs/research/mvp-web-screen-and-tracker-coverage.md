# MVP Web-Screen and Tracker Coverage

Research asset for [Audit complete MVP web-screen and tracker coverage](https://github.com/rafaelromao/slotmerge/issues/272).

## Scope and evidence standard

This audit is a research snapshot of the repository at commit `c13631d9` (`fix(ui): address review findings on PR #269`) on the current branch. It uses the local canonical documents, source, tests, and git history, plus first-party GitHub issue and pull-request records fetched with `gh`.

The target is the complete web delivery described by `docs/mvp-spec.md` section 4, not only the domain and persistence implementation. The glossary in `CONTEXT.md` is authoritative: this artifact uses User, Organizer, Admin, Availability, Availability Window, Calendar Connection, Topic, Topic Proposal, Slot, Search, Search Result, Match, and Discoverability in their defined senses.

Evidence labels used below:

- **Rendered page**: a Next.js page or an HTTP response that emits user-visible HTML.
- **Rendered component**: a React component or HTML renderer tested in isolation, not a browser journey.
- **Handler/API**: an HTTP route or handler returning JSON, redirects, or standalone HTML without a browser.
- **Domain seam**: a repository, service, matching, worker, or provider boundary exercised without rendering the complete screen.
- **Browser evidence**: a test that starts the web app and drives it through a browser. No such harness is configured in this repository as audited.

## Executive findings

1. The implementation has meaningful domain and HTTP coverage, but it is not complete web-screen coverage. The only Next.js product page at `/` is a local runtime scaffold (`app/page.tsx:1-8`); the only substantial Next.js product page is `/searches/[id]/results` (`app/searches/[id]/results/page.tsx:7-78`). `/me/topics` and the five Admin routes emit standalone HTML from route handlers, while most other product surfaces expose JSON only (`app/me/topics/route.ts:11-45`, `src/admin/page.ts:49-58`).
2. The role-aware shell required by the prototype is absent. The root layout only emits `<html>`, `<body>`, and children (`app/layout.tsx:9-14`), while the prototype requires one primary shell, role-aware navigation, `Home`, `My availability`, `Search`, `Search history`, and `Admin` surfaces (`docs/prototypes/core-search-workflow.md:7-17`).
3. The Search Result calendar and Slot Details Drawer are the strongest rendered surfaces. They render server-fetched Search Result snapshots and support slot selection, stale markers, and a drawer (`app/searches/[id]/results/page.tsx:44-76`, `app/searches/[id]/results/SearchResultClient.tsx:132-261`). They still do not provide the complete Screen 4.7-to-4.10 journey: there is no rendered Search form, no week navigation, no rendered Search history page, and no rerun action.
4. Many tests named `E2E` are integration tests around direct function calls. The E2E configuration includes only `tests/e2e/**/*.test.ts` in Vitest, with no browser project or browser runner (`vitest.e2e.config.ts:1-17`). Representative tests import route functions and call them with `new Request`, then inspect JSON or HTML strings (`tests/e2e/organizer-search-renders-weekly-calendar.test.ts:11-16,141-199`, `tests/e2e/admin-invites-user-from-admin-users-screen.test.ts:147-203`).
5. The component tests do render HTML, but only in `renderToString` or `happy-dom`; they do not start Next.js, resolve a route, exercise a real session, or prove a role journey (`tests/searches/results.test.tsx:1-7,38-72`, `tests/slot-details-drawer.test.tsx:1-7,132-231`, `package.json:40-58`).
6. The tracker has closed substantial domain/API tickets whose titles use user-visible verbs, while the canonical completion gates did not require a browser-rendered surface. The original Wayfinder map explicitly stopped at the implementation-ready spec and excluded building the app (`Wayfinder: Topic-aware group availability MVP spec`, [issue #1](https://github.com/rafaelromao/slotmerge/issues/1)). The new map records this failure mode but leaves the route hierarchy, browser gates, and screen ownership unresolved (`Wayfinder: Complete SlotMerge MVP web app implementation plan`, [issue #271](https://github.com/rafaelromao/slotmerge/issues/271)).
7. The first repair target is the canonical contract, not another implementation ticket: every screen needs a named route owner, a rendered behavior contract, a domain/HTTP seam, a browser acceptance owner, and closure evidence that cannot be satisfied by JSON or direct handler invocation alone.

## Screen audit

### 4.1 Invite and Magic Link

**Required rendered behavior.** The screen is an email-only login flow with mandatory transactional magic-link email and a clear statement that Calendar Connection access is separate and free/busy-only (`docs/mvp-spec.md:110-114`). The prototype shows the invited email, assigned role, a Send magic link action, and the separate Calendar Connection explanation (`docs/prototypes/core-search-workflow.md:19-38`). The inherited flow requires an Admin invite, an invitation email, single-use magic-link authentication, and landing on setup (`docs/mvp-spec.md:47-52`).

**Current surface and exact route.** Admin invite HTML exists at `GET/POST /admin/invites`: the GET handler renders an email field, role selector, invite list, and CSRF field, and the POST handler validates, creates the invite, sends an invite email, and redirects (`src/admin/invites.ts:63-79,81-192,226-277`). The magic-link verification URL `GET /auth/magic-link/verify?token=...` emits an auto-submit confirmation document (`src/auth/magic-link-verify.ts:78-95,193-210`); `POST /auth/magic-link/verify` emits HTML error documents for invalid, expired, revoked, or reused links and redirects successful authentication to `/` (`src/auth/magic-link-verify.ts:97-189,213-243`). `POST /auth/magic-link/request` is an action-only form-data handler returning JSON, not an HTML login screen (`app/auth/magic-link/request/route.ts:1-10`, `src/auth/magic-link-request.ts:69-125,201-205`). `POST /auth/magic-link/resend` can emit standalone HTML after an expired link (`src/auth/magic-link-resend.ts:57-163,235-266`).

**HTTP and domain seam.** The HTTP seam is split between the Admin invite handler, magic-link request handler, verify handler, resend handler, session repository, invite repository, and Email delivery service (`src/admin/invites.ts:30-36,124-171`, `src/auth/magic-link-request.ts:46-55,92-124`, `src/auth/magic-link-verify.ts:120-189`). Successful verification creates a session, accepts the invite, seals the session cookie, and redirects to `/` (`src/auth/magic-link-verify.ts:157-188`).

**Tests and what they prove.** Auth tests call request, verify, resend, or token functions directly and inspect response status, cookies, email records, or HTML text; they do not use a browser (`tests/e2e/auth-magic-link-reject-after-expiration.test.ts:109-183`, `tests/e2e/auth-magic-link-request-new-magic-link-after-expired-link.test.ts:121-230`, `tests/e2e/invite-user-role-and-sign-in.test.ts:150-282`). The Admin invite test is named “from the Admin Users screen,” but imports `createAdminInvitesHandlers`, invokes handler functions, extracts a CSRF token from returned text, and submits another direct `Request` (`tests/e2e/admin-invites-user-from-admin-users-screen.test.ts:106-203`).

**Tracker ownership, missing work, and contradictions.** Domain/API ownership is in [Admin invites with email and role](https://github.com/rafaelromao/slotmerge/issues/21), [Send invitation email with magic link](https://github.com/rafaelromao/slotmerge/issues/22), [Sign in via magic link](https://github.com/rafaelromao/slotmerge/issues/23), [Request new magic link after invalid or expired link](https://github.com/rafaelromao/slotmerge/issues/24), [Reject sign-in for non-invited email](https://github.com/rafaelromao/slotmerge/issues/25), and [Sub-PRD: Auth & Invites](https://github.com/rafaelromao/slotmerge/issues/16). Related merged PRs are [Admin invites a User with email and role](https://github.com/rafaelromao/slotmerge/pull/146), [feat: send invitation email with magic link](https://github.com/rafaelromao/slotmerge/pull/153), [feat: sign in via magic link](https://github.com/rafaelromao/slotmerge/pull/162), [feat: reject sign-in for non-invited email](https://github.com/rafaelromao/slotmerge/pull/167), and [feat: request new magic link after invalid or expired link](https://github.com/rafaelromao/slotmerge/pull/168). E2E ownership includes [Invite role selection is explicit for Organizer and Admin](https://github.com/rafaelromao/slotmerge/pull/199), [invite User role and verify single-use magic link acceptance](https://github.com/rafaelromao/slotmerge/pull/206), [magic link is rejected after use](https://github.com/rafaelromao/slotmerge/pull/207), [magic link is rejected after expiration](https://github.com/rafaelromao/slotmerge/pull/208), [request new magic link after expired link](https://github.com/rafaelromao/slotmerge/pull/220), and [Admin invites a User with role selection from the Admin Users screen](https://github.com/rafaelromao/slotmerge/pull/233).

### 4.2 Setup Checklist Home

**Required rendered behavior.** The Home lists Profile, Discoverability consent, Topics, Availability, and optional Calendar Connection progress, and continues until required setup is complete (`docs/mvp-spec.md:116-120`). Setup completion requires display name, Discoverability consent, at least one Topic or Topic Proposal, and an Availability source or manual Availability Window (`docs/mvp-spec.md:36-43`). The prototype shows a checklist, readiness explanation, and Continue setup action (`docs/prototypes/core-search-workflow.md:40-62`).

**Current surface and exact route.** `GET /` renders only two static lines and no checklist (`app/page.tsx:1-8`). `GET /me` returns the profile, setup item list, Discoverability state, Topic and Topic Proposal state, Availability references, Calendar Connection references, and computed Search eligibility as JSON (`app/me/route.ts:220-255,317-366`). The setup computation correctly models four required items and one optional Calendar Connection (`app/me/route.ts:42-95`). There is no `/setup` page and no rendered checklist route.

**HTTP and domain seam.** The HTTP seam is `GET/PATCH/DELETE /me`; the domain seam reads the profile and related state, computes setup completeness, computes eligibility, updates profile attributes, and deletes the profile (`app/me/route.ts:220-366,383-407`). The matching eligibility predicate requires consent, active status, display name, a Topic or Topic Proposal, and an Availability source (`src/search/search-snapshot-assembler.ts:259-287`).

**Tests and what they prove.** `tests/app-shell.test.ts:1-14` calls the `Home` component function and asserts the scaffold text; it does not render a route or browser page. `tests/e2e/setup-checklist-gates-matching-eligibility.test.ts:32-75,77-175` calls `SearchSnapshotAssembler` directly and checks matching eligibility; it never reads setup HTML. `tests/e2e/user-sets-profile-attributes.test.ts:3-7,71-97,139-193` calls `/me` handlers and asserts JSON/database state. `tests/e2e/list-pending-topic-proposals-on-profile.test.ts:174-197` checks setup JSON for a pending Topic Proposal, not the Home screen.

**Tracker ownership, missing work, and contradictions.** Ownership is [Show setup checklist and gate setup completion](https://github.com/rafaelromao/slotmerge/issues/36), [E2E test: setup checklist gates matching eligibility](https://github.com/rafaelromao/slotmerge/issues/87), [feat: show setup checklist and gate setup completion](https://github.com/rafaelromao/slotmerge/pull/149), and [E2E test: setup checklist gates matching eligibility](https://github.com/rafaelromao/slotmerge/pull/198), under [Sub-PRD: Profile & Setup](https://github.com/rafaelromao/slotmerge/issues/19). The original app-shell feature was [Provision app shell, auth, and Postgres bootstrap](https://github.com/rafaelromao/slotmerge/issues/20).

The JSON setup state is substantial, but the canonical screen requirement is not met. The spec and PRD describe a setup Home while code and tests close on `/me` JSON and matching eligibility. The current root page’s scaffold text is explicit evidence that the screen was not implemented (`app/page.tsx:3-6`).

### 4.3 Discoverability Consent

**Required rendered behavior.** The screen must explain what Organizers may see and what they will not see, require explicit consent, and make the privacy boundary understandable (`docs/mvp-spec.md:121-124`; `docs/prototypes/core-search-workflow.md:64-90`). Consent controls whether a User appears in matching Search Results (`docs/mvp-spec.md:426-430`).

**Current surface and exact route.** There is no rendered consent page. `POST /me/discoverability-consent` accepts JSON `{ confirmed: true }` and returns JSON consent state; `DELETE /me/discoverability-consent` revokes it and returns JSON (`app/me/discoverability-consent/route.ts:17-67`). `GET /me` includes consent state and Search eligibility in JSON (`app/me/route.ts:326-366`).

**HTTP and domain seam.** The handler validates CSRF and calls `grantDiscoverabilityConsent` or `revokeDiscoverabilityConsent` (`app/me/discoverability-consent/route.ts:17-77`). The matching seam reads the persisted consent record and rejects a User without it (`src/search/search-snapshot-assembler.ts:165-190,266-287`).

**Tests and what they prove.** `tests/discoverability-consent-route.test.ts:1-3` and the route tests exercise direct POST/DELETE handlers. [Grant discoverability consent makes User eligible for matching](https://github.com/rafaelromao/slotmerge/pull/219) and [revoke discoverability removes User from matching](https://github.com/rafaelromao/slotmerge/pull/217) test route/domain effects and Search membership; they do not render the consent explanation or checkbox in a browser. The E2E test inventory identifies these journeys as direct route/domain tests (`tests/e2e/grant-discoverability-consent-makes-user-eligible-for-matching.test.ts:126`, `tests/e2e/revoke-discoverability-removes-user-from-matching.test.ts:202`).

**Tracker ownership, missing work, and contradictions.** Ownership is [Grant and revoke discoverability consent](https://github.com/rafaelromao/slotmerge/issues/28), [E2E test: grant discoverability consent makes user eligible for matching](https://github.com/rafaelromao/slotmerge/issues/76), [E2E test: revoke discoverability removes user from matching](https://github.com/rafaelromao/slotmerge/issues/77), [Grant and revoke discoverability consent](https://github.com/rafaelromao/slotmerge/pull/154), [E2E test: grant discoverability consent makes user eligible for matching](https://github.com/rafaelromao/slotmerge/pull/219), and [E2E test: revoke discoverability removes user from matching](https://github.com/rafaelromao/slotmerge/pull/217), under [Sub-PRD: Profile & Setup](https://github.com/rafaelromao/slotmerge/issues/19).

Missing work is the explanation, explicit checkbox/action, consent success/error states, revoke control in the later User surface, shared shell, and browser acceptance. The code-level privacy rule is stronger than the rendered evidence: matching enforces consent, but no rendered page explains the boundary.

## Root cause of the API-only false completion

The false completion was a planning and evidence failure, not a lack of domain work.

1. The closed original Wayfinder map’s destination was an implementation-ready specification, and its closing comment says the spec was committed. Its out-of-scope list explicitly excluded building the app (`Wayfinder: Topic-aware group availability MVP spec`, [issue #1](https://github.com/rafaelromao/slotmerge/issues/1)). The downstream implementation tickets therefore inherited domain/API-shaped acceptance without a screen-level completion gate.
2. The top-level PRD declared a “single user-visible Organizer Search Result page” as the primary E2E seam and said a good test observes rendered HTML, but it did not make a real browser or a named route a closure requirement ([SlotMerge MVP PRD](https://github.com/rafaelromao/slotmerge/issues/14), Implementation Decisions and Testing Decisions). The same PRD says the E2E plan maps acceptance criteria to test cases at the Search Result page seam, which is not true for setup, Calendar Connection, Admin, or auth journeys under the current code.
3. The E2E plan explicitly promised “HTTP responses, rendered HTML, persisted records via API, mock email/webhook events” and a `searchResultPage` helper that inspects the rendered calendar and clicked drawer (`E2E test plan: SlotMerge MVP`, [issue #62](https://github.com/rafaelromao/slotmerge/issues/62)). The resolved decisions chose Vitest and said E2E tests are not executed in CI, but did not resolve what “browser” meant or require a browser runner.
4. The repository’s E2E configuration makes the gap concrete: it includes only TypeScript test files, configures database global setup and setup files, and has no browser project (`vitest.e2e.config.ts:1-17`). The package has `happy-dom` and `@testing-library/react`, but no browser automation dependency (`package.json:40-58`).
5. User-visible tracker titles were accepted with direct handler/domain evidence. The Search test titled “run a Search and render weekly calendar result” imports `GET` from `/api/searches/[id]`, calls `submitSearch`, and asserts JSON (`tests/e2e/organizer-search-renders-weekly-calendar.test.ts:11-16,141-199,201-359`). The Search history test titled “Search history is visible…” also imports the JSON API and compares payloads (`tests/e2e/search-history-visible-to-all-organizers-and-immutable.test.ts:12-27,168-180,200-451`). The Admin invite test titled “from the Admin Users screen” calls the handler factory and reads HTML strings (`tests/e2e/admin-invites-user-from-admin-users-screen.test.ts:3-8,147-203`).
6. The UI audit PR corrected the Search Result component baseline, but its own scope explicitly deferred “new UI surfaces outside Search Result + Drawer” and left `/` as a foundation page ([PR #269](https://github.com/rafaelromao/slotmerge/pull/269)). That PR can prove improved component rendering; it cannot prove the 13-screen MVP.
7. Git history shows the sequence: domain/API and handler tests were merged before the UI audit baseline (`git log` entries `f67b8505`, `b1019b41`, `0104d0f9`, `91b33cc9`, followed by `e6853cd2` and `c13631d9`). The tracker used “render,” “screen,” and “E2E” in ticket titles without a shared evidence taxonomy distinguishing browser navigation, handler HTML, component HTML, JSON, and domain seams.

The root correction is therefore a completion contract: a JSON response can close an API seam; a direct function test can close a domain seam; neither can close a rendered screen or role journey. Each later ticket needs to name the route, the rendered states, the browser journey, and the exact evidence artifact before it can be marked complete.

- “Tests assert observable behavior only: HTTP responses, rendered HTML, persisted records via API, mock email/webhook events” must define which tests are browser tests and which are HTTP/domain tests. A returned HTML string from a handler and `renderToString` are not browser evidence.
- The promised `searchResultPage` helper that “runs an Organizer Search end-to-end and inspects the rendered calendar and clicked Slot details drawer payload” is not present in the audited tree. The actual Search E2E test calls `submitSearch`, imports an API GET handler, and checks JSON (`tests/e2e/organizer-search-renders-weekly-calendar.test.ts:11-16,141-199`). Either implement the helper later or remove the promise and replace it with an explicit browser-harness ticket.
- The coverage map says stories map to tests, but it does not map each screen and role journey to a canonical route and browser test. Add a screen/role/route/evidence matrix covering 4.1–4.13 and flows 3.1–3.8.
- Vitest remains the locked test framework, but the resolved comment only says E2E is not run in CI and does not choose a browser mechanism ([issue #62 comment](https://github.com/rafaelromao/slotmerge/issues/62#issuecomment-4940253277)). This must be explicitly coordinated with [Research real-browser acceptance and mocked demo options](https://github.com/rafaelromao/slotmerge/issues/273) and [Choose browser acceptance, demo, and capture gates](https://github.com/rafaelromao/slotmerge/issues/274).
- “Mocks only external services” should be preserved, but the plan must state that the web app itself is started and driven through canonical routes, with provider/Email/clock mocks at the locked boundaries (`AGENTS.md:21-25`, `vitest.e2e.config.ts:12-17`).

## Planning guardrails

These are constraints for later planning, not implementation decisions:

1. Preserve the locked architecture and glossary. Do not replace the single full-stack web app, PostgreSQL, DB-backed job queue, internal matching, immutable Search Result snapshot, encrypted Calendar Connection tokens, or separate Calendar OAuth (`AGENTS.md:19-25`).
2. Treat every section 4 item as a web-screen requirement. A ticket is not complete when its JSON route, repository, worker, or component test passes unless the ticket explicitly owns only that seam.
3. Every screen ticket must name the canonical route, page/handler owner, domain and HTTP seam, role visibility, rendered success/error/empty/stale/reconnect states, accessibility/responsive requirements, and browser evidence.
4. Keep direct module tests for matching, timezone, Availability, similarity blocking, token encryption, provider behavior, repositories, workers, and Email delivery. Do not promote those tests into browser evidence.
5. Keep HTTP/handler tests for auth callbacks, form action validation, provider webhooks, and Admin/Calendar Connection seams. Add browser tests on top for user-visible journeys; do not replace narrow seam tests with a browser test.
6. Use the single global clock and locked provider/Email mock boundaries. Avoid tests that sleep or call live providers (`AGENTS.md:23-24`, `docs/mvp-spec.md:324-329`).
7. Keep Search Result snapshots immutable and preserve the display-only boundary. No later screen ticket may add invitations, RSVP, booking, reservation, event creation, notification inbox, or export/share handoff aids (`PRODUCT.md:17-21`, `docs/mvp-spec.md:355-366`).
8. Do not infer route ownership from a tracker title. The closure evidence must link the actual route and test; “render,” “screen,” and “E2E” must be tagged as browser, rendered component, handler HTML, HTTP/API, or domain.
9. Do not close the parent PRD or Wayfinder child from JSON/domain evidence alone. Parent closure should require a complete screen/role matrix, browser journey results, and explicit accounting for every exception.
10. Repair the canonical docs and tracker contract before creating implementation slices that repeat the current route/API ambiguity. The repair work is already represented by [Repair canonical MVP plan and implementation dependency graph](https://github.com/rafaelromao/slotmerge/issues/277).

## Uncertainty and boundary of this audit

The repository has not yet chosen or configured a real-browser runner, so this audit does not claim that a browser test is impossible; it claims only that no browser test configuration or dependency is present in the audited tree. The exact browser option, route migration strategy, visual-capture policy, and final screen hierarchy are intentionally left to the open Wayfinder tickets above. No GitHub issue was modified by this research.
